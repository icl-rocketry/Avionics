In previous versions, the copy constructor of {\ttfamily Basic\+Json\+Document} looked at the source\textquotesingle{}s {\ttfamily memory\+Usage()} to choose its capacity. Now, the copy constructor of {\ttfamily Basic\+Json\+Document} uses the same capacity as the source.

Example\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{ \{c++\}}
\DoxyCodeLine{DynamicJsonDocument doc1(64);}
\DoxyCodeLine{doc1.set(String("{}example"{}));}
\DoxyCodeLine{}
\DoxyCodeLine{DynamicJsonDocument doc2 = doc1;}
\DoxyCodeLine{Serial.print(doc2.capacity());  // 8 with ArduinoJson 6.14}
\DoxyCodeLine{                                // 64 with ArduinoJson 6.15}

\end{DoxyCode}


I made this change to get consistent results between copy-\/constructor and move-\/constructor, and whether RVO applies or not.

\hypertarget{md___users_kiran__documents__projects__i_c_l_r__avionics__ricardo__o_s__ricardo__o_s_lib__arduino_json_6_x__c_h_a_n_g_e_l_o_g_autotoc_md23}{}\doxysubsubsection{Na\+N and Infinity}\label{md___users_kiran__documents__projects__i_c_l_r__avionics__ricardo__o_s__ricardo__o_s_lib__arduino_json_6_x__c_h_a_n_g_e_l_o_g_autotoc_md23}
The JSON specification allows neither NaN not Infinity, but previous versions of Arduino\+Json supported it. Now, Arduino\+Json behaves like most other libraries\+: a NaN or and Infinity in the {\ttfamily Json\+Document}, becomes a {\ttfamily null} in the output JSON. Also, {\ttfamily deserialize\+Json()} returns {\ttfamily Invalid\+Input} if the JSON document contains NaN or Infinity.

This version still supports NaN and Infinity in JSON documents, but it\textquotesingle{}s disabled by default to be compatible with other JSON parsers. {\bfseries{Personal note about the {\ttfamily \mbox{\hyperlink{class_string}{String}}} class}}\+: Support of the {\ttfamily \mbox{\hyperlink{class_string}{String}}} class has been added to the library because many people use it in their programs. However, you should not see this as an invitation to use the {\ttfamily \mbox{\hyperlink{class_string}{String}}} class. The {\ttfamily \mbox{\hyperlink{class_string}{String}}} class is {\bfseries{bad}} because it uses dynamic memory allocation. Compared to static allocation, it compiles to a bigger, slower program, and is less predictable. You certainly don\textquotesingle{}t want that in an embedded environment!
